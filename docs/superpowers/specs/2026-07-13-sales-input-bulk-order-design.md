# 영업 입력 화면 대량 오더 등록 개선

날짜: 2026-07-13
대상 파일: `sales-input.jsx` (`SalesInputScreen`의 "제품 선택" 섹션, 신규 등록 모드 한정)

## 배경

영업 담당자가 한 오더에 여러 제품(모델)을 대량 등록할 때 아래 세 가지 불편이 있다.

1. "행 추가" 버튼이 한 번에 한 행만 추가한다. 20개 모델을 등록하려면 버튼을 20번 눌러야 한다.
2. 모델 선택이 행마다 모달(`ModelSelectModal`)을 열고 충전속도 → 모델을 다시 고르는 과정을 반복해야 한다. 같은 모델을 여러 행에 반복 지정할 때 특히 번거롭다.
3. 모델·용도·CPO 운영사처럼 여러 행에서 같은 값을 쓰는 필드를 한 번에 채우는 방법이 없다. 행 복제(`duplicateRow`)는 있지만 1행씩만 복제된다.

이 프로젝트에는 이미 `production-waiting.jsx`의 "일괄 처리" 플로우(체크박스 다중선택 + 액션바)가 동일한 종류의 문제(여러 오더에 같은 작업을 반복 적용)를 해결하고 있다. 이번 개선은 그 UI 언어를 신규 오더 입력 폼의 "제품 선택" 테이블에도 그대로 적용한다.

## 범위

- 대상: `SalesInputScreen`의 제품 선택 테이블, `!isEdit`(신규 등록) 모드에서만 동작. 수정 모드(`isEdit`)는 행이 항상 1개이므로 이번 변경의 영향을 받지 않는다.
- DB 스키마 변경 없음. 클라이언트 폼 상태(`rows`)만 다룬다.
- 기존 개별 조작(행별 모델 선택 모달, 행 복제, 행별 삭제, 용도 chip 토글)은 그대로 유지한다. 이번 기능은 대체가 아니라 추가 옵션이다.

## 설계

### 1. 행 식별자(`_id`) 도입

현재 각 행은 배열 인덱스로만 식별되며 React key도 `key={i}`를 쓴다. 체크박스 선택 상태(`selectedRowIds`)를 행 삭제·추가와 무관하게 안정적으로 유지하려면 인덱스가 아닌 고유 id가 필요하다.

- 컴포넌트 내부에 `useRefSI` 기반 카운터(`nextRowIdRef`)를 두고, `makeRow()`와 `duplicateRow(i)`가 행을 새로 만들 때마다 `_id: nextRowIdRef.current++`를 부여한다.
- `rows.map((row, i) => ...)`의 React `key`를 `row._id`로 변경한다 (현재 `key={i}`).
- `selectedRowIds`는 `Set<number>`(행 `_id` 집합)으로 관리한다. 인덱스 기반 상태가 아니므로 행이 삭제/추가돼도 다른 행의 선택 여부가 바뀌지 않는다.
- `_id`는 다른 `_power`처럼 제출 시 payload에서 제외되는 내부 필드다 (`submit()`의 구조 분해 시 함께 제거).

### 2. 체크박스 다중선택

- 표 맨 왼쪽에 체크박스 열을 추가한다 (`!isEdit`일 때만 렌더링).
- 헤더: 전체선택/해제 체크박스. `production-waiting.jsx`의 `TableGroup` 헤더 체크박스와 동일한 패턴 — 일부만 선택된 상태는 `indeterminate`로 표시 (`useEffect`로 `ref.current.indeterminate` 설정).
- 각 행: 개별 체크박스, `aria-label={`${i + 1}번째 행 선택`}`, 클릭 시 이벤트 버블링 방지(`stopPropagation`, 기존 행별 삭제/복제 버튼과 같은 열이므로 오작동 방지).
- 선택 토글 함수: `toggleRowSelect(id)`, `toggleAllRows()`.

### 3. 일괄 적용 툴바

선택된 행이 1개 이상이면 표 바로 위(카드 바디 상단)에 `toolbar` 클래스(기존 스타일, `production-waiting.jsx`의 일괄 처리 바와 동일한 배경색 `var(--primary-50)` / 테두리 `var(--primary-100)`)로 액션바를 렌더링한다.

내용:
- 좌측: `{selectedRowIds.size}행 선택됨`
- **모델 일괄 지정** 버튼: 클릭 시 기존 `ModelSelectModal`을 연다. 기존에는 `modelModalRow`(단일 행 인덱스)를 모달에 넘겼지만, 일괄 지정 시에는 `modelModalRow`를 특수값(예: `'bulk'`)으로 설정해 모달이 "선택된 모든 행에 적용" 모드로 동작하게 한다. `onSelect(model, power)` 콜백에서 `modelModalRow === 'bulk'`이면 `selectedRowIds`에 속한 모든 행의 `model_name`/`_power`를 갱신하고, 아니면 기존처럼 해당 인덱스 행만 갱신한다.
- **용도** 공용/비공용 chip: 클릭 즉시 선택된 모든 행에 적용. 비공용으로 전환되는 행은 기존 행별 토글 로직과 동일하게 `cpo_name`/`station_id`/`charger_no`를 초기화한다.
- **CPO 운영사** 입력: `BulkInlineCombo` + "적용" 버튼. 적용 시 선택된 행 중 `usage_type === '공용'`인 행에만 `cpo_name`을 갱신한다. 선택된 행 중 공용 행이 하나도 없으면 CPO 입력·적용 버튼을 비활성화하고 `title="선택한 행 중 공용 용도가 없습니다"`를 표시한다.
- **선택 삭제** 버튼: `window.actions.showConfirm`으로 확인 후 선택된 모든 행을 제거. 단, 선택된 행 수가 전체 행 수와 같으면(모두 선택) 최소 1행은 남아야 하므로 버튼을 비활성화하고 `title="최소 1개 행은 남아야 합니다"`를 표시한다.
- **선택 해제** 버튼: `selectedRowIds`를 빈 Set으로 초기화.
- 충전소ID·충전기번호·라우터번호·USIM번호는 충전기별 고유값이라 일괄 적용 대상에서 제외한다 (의도적 제외 — 동일 값 다중 등록으로 인한 중복 데이터 방지).

### 4. N행 한번에 추가

- "행 추가" 버튼 좌측에 숫자 입력(`type="number"`, 기본값 1, `min=1 max=50`)을 추가한다.
- `addRow()`를 `addRow(count = 1)`로 변경: 마지막 행이 비공용이면 새로 추가되는 `count`개 행 모두 비공용으로 시작하는 기존 규칙을 유지하면서, `_id`를 각각 새로 발급해 `count`개 행을 한 번에 append한다.
- 버튼 라벨은 입력값에 따라 동적으로 바뀐다 (예: `N행 추가`). 입력값이 1이면 기존과 동일한 문구/동작을 유지한다.

### 5. 기존 로직과의 상호작용

- `submit()` 성공 후 폼 초기화(`setRows([makeRow()])`) 시 `selectedRowIds`도 함께 빈 Set으로 초기화한다.
- "초기화" 버튼 클릭 시에도 동일하게 `selectedRowIds`를 초기화한다.
- `removeRow(i)`(개별 삭제)로 행이 사라질 때 해당 `_id`가 `selectedRowIds`에 남아있으면 함께 정리한다 (다음 렌더에서 존재하지 않는 `_id`가 선택된 것처럼 보이지 않도록).
- `rowErrors`/`validRows`/`totalValidQty` 등 기존 검증·집계 로직은 변경하지 않는다. 일괄 지정도 결국 각 행의 필드값을 바꾸는 것뿐이므로 기존 파이프라인을 그대로 통과한다.

## 검증 계획

- `npm test` 통과 확인 (역할별 권한 테스트라 이번 변경과는 무관하지만 회귀 없는지 확인).
- 브라우저에서 수동 확인 (자동화 테스트 없는 코드베이스):
  - 행 5개를 한 번에 추가 → 5행이 실제로 추가되는지
  - 체크박스로 3행 선택 → 모델 일괄 지정 → 3행 모두 모델이 바뀌는지, 나머지 행은 그대로인지
  - 용도 일괄 전환(공용→비공용) → 선택된 행의 CPO/충전소ID/충전기번호가 초기화되는지
  - CPO 일괄 적용 → 선택 행 중 비공용 행은 영향받지 않는지
  - 전체 선택 후 "선택 삭제" 버튼이 비활성화되는지, 1행 남기고 선택 시 정상 삭제되는지
  - 개별 행 삭제 후에도 나머지 선택 상태가 유지되는지 (인덱스가 아닌 `_id` 기반이므로)
  - 제출 후 폼이 초기화되며 선택 상태도 함께 초기화되는지
  - 수정 모드(`isEdit`)에서는 체크박스 열과 일괄 툴바가 아예 보이지 않는지
